\section{Λογίσμικα που χρησιμοποιήθηκαν για την κατασκευή του server}

\subsection{Flask}

Για την ανάπτυξη του (διακομιστή ιστού) web server της IoT εφαρμογής, χρησιμοποιήσαμε την Flask.
Η Flash αποτελεί ένα ελαφρό framework της γλώσσας προγραμματισμού python 
για την δημιουργία web εφαρμογών, χρησιμοποιώντας το πρωτόκολλο Web Server Gateway Interface (WSGI) \cite{Flask}.
Η Flask αποτελεί μία από τις πιο δημοφιλές επιλογές framework για ανάπτυξη στον Ιστό,
μαζί με την FastAPI και την Django\cite{jetbrainsPopular}.

Το WSGI, όπως έχει οριστεί από την Python Enhancement Proposals 3333,αποτελεί ένα δημοφιλές πρωτόκολλο της python για την διασύνδεση μεταξύ web servers και web εφαρμογών \cite{PEP3333}.
Για την λειτουργία της ,η Flask εξαρτάται από το WSGI toolkit και the Jinja template engine για την δημιουργία περιγραμμάτων (templates)
που διαμορφώνουν την μορφή της html στις σελίδων που επιστρέφει ο server στον (πελάτη) client,την οποία και χρησιμοποιούμε για την διαμόρφωση της ιστοσελίδας.
Για server, χρησιμοποιήσαμε το Waitress έκδοσης 3.0.2 \cite{waitress},που είναι βασισμένο πλήρως στην python και στο πρότυπο WSGI.

Δεν θα περιγράψουμε ρητά πως αναπτύχθηκε η web εφαρμογή που διαχειρίζεται την http αίτηση, όπως κάναμε με τον μικροελεκτή. 
Η λειτουργία της web εφαρμογής θα φανεί σε σχέση με την αλληλεπίδραση του server με τα άλλα προγράμματα που θα περιγράψουμε στην πορεία,
όπως και με την αλληλεπίδραση του χρήστη με τον server.

Για την ιστοσελίδα στην πλευρά του χρήστη (front-end),δεν έχει χρησιμοποιηθεί κάποιο άλλο framework.
Χρησιμοποιούμε Vanilla Javascript και CSS

\subsection{MongoDB}

Για την αποθήκευση και διαχείριση των δεδομένων που λαμβάνει ο server της εφαρμογής, χρησιμοποιήσαμε την MongoDB Community Edition,
που αποτελεί την δωρεάν μορφή της MongoDB.
Η MongoDB αποτελεί μια βάση δεδομένων εγγράφων (documents database) που λειτουργεί σε πραγματικό χρόνο (Operational database).
Αποθηκεύει τα δεδομένα τις σε ευέλικτης δομής, JSON τύπου, documents(έγγραφα),το οποίο κάνει εύκολο την μοντελοποίηση δεδομένων με τον ίδιο τρόπο 
που τα χρησιμοποιεί και ο κώδικας των εφαρμογών που θα χρησιμοποιήσει τα δεδομένα.
Δεν επιβάλλεται αυστηρή μορφή (schema) που πρέπει να έχουν τα δεδομένα, οπότε μπορούμε να διαχειριστούμε μαζί documents με διαφορετική μορφή
και μπορούμε να αλλάξουμε την μορφή που έχουν τα document χωρίς πρόβλημα.

Ακόμα, με την τεχνική Sharding δίνεται η δυνατότητα επέκτασης της βάσης δεδομένων σε πολλαπλά μηχανήματα,
δηλαδή οριζόντια επέκταση (Horizontal scaling),απελευθερώνοντάς μας από την ανάγκη του να προσθέσουμε παραπάνω υπολογιστικούς πόρους σε ένα μηχάνημα, στην περίπτωση που θέλαμε να επεκτείνουμε τις δυνατότητες της βάσης δεδομένων,
δηλαδή Vertical Scaling \cite{mongoDBoverview}.

Τα documents αποθηκεύονται με μορφή BSON, που είναι μια binary μορφή των json αντικειμένων που υποστηρίζει παραπάνω βασικούς τύπους δεδομένων από ότι τα json αντικείμενα.
Τα documents αποτελούνται από ζευγάρια πεδίων-τιμών, με τα πεδία να είναι πάντα τύπου String, δηλαδή απλώς κείμενο.
Οι τιμές να είναι οποιοσδήποτε από τους βασικούς τύπους BSON, όπως είναι οι τύπου String, Boolean (λογικοί τύποι),  Date (ημερομηνία) και άλλα,
αλλά και άλλα εμφολευμένο (nested  ) document ,πίνακες και πίνακες από documnets.

Κάθε βάση δεδομένων (database),που ονομάζεται database στην Mongodb, χωρίζεται σε συλλογές (collection),
με τα collection να κατέχουν μέσα τους τα documents.
Τα collection ορίζονται περισσότερο για ομαδοποίηση documents με παρόμοια ταυτότητα.
Σε κάθε collection ενός database δίνεται ένα αμετάβλητο (immutable), χωρίς δυνατότητα τροποποίησης μετά από   την ανάθεση της τιμής,
μοναδικό αναγνωριστικό (Universally  unique identifier ,UUID) για να μπορεί να αναγνωριστεί μεταξύ των αντιγράφων του set στα διαφορετικά μηχανήματα
που μπορεί να βρίσκεται το collection.

Δεν επιβάλλεται κάποιου είδος (schema) στα document.
Όμως, σε κάθε document ενός collection, όχι nested document, απαιτείται να έχει ένα πεδίο που ονομάζεται \texttt{\_id},
το οποίο έχει μοναδική τιμή για το συγκεκριμένο collection, η οποία δεν επαναλαμβάνεται η τιμή αυτή για πεδίο \texttt{\_id} κάπου αλλού στο collection
To \texttt{\_id} είναι το κύριο κλειδί του document, δηλαδή το αναγνωριστικό του για να ξεχωρίζει σε σχέση με τα υπόλοιπα documents.
Εάν δεν δοθεί πεδίο με όνομα \texttt{\_id}, τότε η ίδια η mongoDB το προσθέτει στο document, δημιουργώντας μια τιμή τύπου ObjectId.

Οι δύο κύριοι λόγοι για τους οποίους επιλέχθηκε η MongoDB είναι ότι κάνει πολύ εύκολη την αποθήκευση των δεδομένων που έρχονται από τις συσκευές μας και κρατάει ανέπαφη την json μορφή των δεδομένων.
Η κάθε δειγματοληψία, μαζί με τις έξτρα πληροφορίες που προστέθηκαν από τους μικροελεκτές το id της συσκευής, δηλαδή του μικροελεκτή μαζί με τον αισθητήρα, την χρονική στιγμή που συλλέχθηκε η δειγματοληψία και το όνομα του αισθητήρα που έκανε την δειγματοληψία, αποτελούν αυτόνομα json αντικείμενα και δεν χρειάζονται να γνωρίζουν πληροφορίες μεταξύ τους.


Για την εφαρμογή μας, χρησιμοποιήθηκαν δύο collection.
Το πρώτο collection, που είναι και το πιο μεγάλο σε όγκο δεδομένων, περιέχει τα δεδομένα που έχουν στείλει οι μικροελεκτές στον server.
Κάθε ξεχωριστό document είναι μία δειγματοληψία που έχει πραγματοποιηθεί.
Η μορφή των documents βασίζονται στην μορφή των δεδομένων όπως τα έστειλε ο μικροελεκτής στο server.
Τα documents από αισθητήρες με το ίδιο όνομα έχουν πάντα την ίδια μορφή δεδομένων

Το δεύτερο collection περιέχει τα δεδομένα που έχουμε πάρει από τον χρήστη, μέσω του client-sided (πλευρά του πελάτη) κομματιού της web εφαρμογής που έχουμε δημιουργήσει.
Είναι δεδομένα που ο χρήστης δηλώνει ρητά στο server, που σκοπό έχουν την συμπλήρωση στοιχείων που θα χρησιμοποιήσουμε για την μετέπειτα επεξεργασία των μετρήσεων που συλλέχθηκαν στα πειράματα.
Το userInputCategory  μας υποδεικνύει τι κατηγορία δεδομένων είναι, αν και μπορεί να υπάρχουν διαφορετικές υποκατηγορίες με βάση την κατηγορία.

Η ανάγκη για διαφορετικές κατηγορίες προέκυψε γιατί είχαμε υλοποιήσει την δυνατότητα για τον χρήστη να εισάγει θερμοκρασία και υγρασία δωματίου, με γνώση που υπήρχε από όργανο εκτός των αισθητήρων.
Οι δύο σκοποί της λειτουργίας αυτής είναι η αποστολή των δεδομένων αυτών σε αισθητήρες που το χρειάζονται για καλύτερο υπολογισμό της ποιότητας του άερα,όπως ήταν ο CCS811,και η συμπερίληψη των περιβαλλοντικών συνθήκων στην μετέπειτα ανάλυση και υλοποίηση των μηχανικών μοντέλων, χωρίς να βασιζόμαστε μόνο στις τιμές του αισθητήρα θερμοκρασία και υγρασίας του ΒΜΕ680.

Δεν χρησιμοποιήθηκε η συγκεκριμένη λειτουργία στα πειράματα, γιατί ο CCS811 που το χρησιμοποίησε σαν πληροφορία ,σταματήσαμε να το χρησιμοποιούμε πριν την έναρξη των πειραμάτων.                                                                                                                                                  Kαι o bme680  δεν χρησιμοποιεί εξωτερικές μετρήσεις, καθώς έχει τους δικούς του αισθητήρες.
Και δεν εντάξαμε τις περιβαλλοντικές συνθήκες, καθώς η BSEC συμπεριλαμβάνει είδη την υγρασία και θερμοκρασία στα μοντέλα που υπολογίζουν την ποιότητα του αέρα.
Μην ξέροντας πως λειτουργούν τα εσωτερικά μοντέλα της BSEC, και αφού δεν προσεγγίσαμε την εύρεση της πηγής ρύπου με κάποιο μοντέλο που περιλαμβάνει υγρασία και θερμοκρασία, δεν έχει νόημα να απαιτούμε από τον χρήστη να συλλέγει την θερμοκρασία και υγρασία, ειδικά στην συχνότητα που απαιτεί τέτοια ανάλυση.
Το όργανο που χρησιμοποιήθηκε για την μέτρηση ποιότητας και υγρασίας δεν είχε κάποιο τρόπο αποστολής δεδομένων στον υπολογιστή.
Παρόλα αυτά, έχει παραμείνει η συγκεκριμένη λειτουργία, σε περίπτωση που μελλοντικά χρειάζεται η γνώση της θερμοκρασίας και υγρασίας από εξωτερικό μετρητή για τους δύο παραπάνω λόγους που αναφέραμε.

Οπότε η βασική κατηγορία δεδομένων για το collection 
σχετικά με τον χρήστη είναι αυτά που στο πεδίο \texttt{UserInputCategory},
 έχουν τιμή \texttt{ExperimentState}.
Στα πειράματα μας, τα δεδομένα αυτά χωρίζονται σε τρεις υποκατηγορίες, με βάση την τιμή που έχουν στο πεδίο \texttt{ExperimentState}.
Οι τρεις υποκατηγορίες είναι οι καταστάσεις του πειράματος \texttt{StartingExperiment},\texttt{InsertingSourcePollutant} και \texttt{RemovingSourcePollutant}.
Η πρώτη και η τρίτη υποκατηγορία έχουν σαν βασικά και μόνο πεδίο που κρατάμε την χρονική στιγμή που δηλώνονται από τον χρήστη.
Η κατάσταση \texttt{InsertingSourcePollutant} εκτός από την χρονική στιγμή, έχει και άλλες λεπτομέρειες που δίνουν στοιχεία για την πηγή του ρύπου και τις συνθήκες του πειράματος.
Οι λεπτομέρειες θα περιγραφτούν όταν μιλήσουμε για την διαδικασία πειραμάτων.

\subsection{Grafana:Για οπτικοποίηση}

Χρησιμοποιήσαμε την Grafana Open Source Software (OSS) για να οπτικοποιήσουμε τις μετρήσεις των αισθητήρων σε ζωντανό χρόνο.
Η Grafana είναι πλατφόρμα με  λογισμικό ανοιχτού κώδικα για την οπτικοποίηση και την ανάλυση δεδομένων.
Επιτρέπει την αναζήτηση(queries), οπτικοποιήση και εξερεύνηση μετρικών δεδομένων, αρχείων καταγραφής (logs) και τα traces, ανεξάρτητα από το πού είναι αποθηκευμένα.
Επίσης, παρέχει εργαλεία για την μετατροπή βάσεων δεδομένων χρονοσειρών (Time Series Databases,TSDB)  σε κατανοητά γραφήματα και οπτικοποιήσεις \cite{grafana}.


Οι χρονοσειρές ορίζονται έως ταξινομημένη ακολουθία τιμών μιας μεταβλητής σε ισαπέχοντα χρονικά διαστήματα \cite{timeSeriesDefinition}.
Οι μετρήσεις των αισθητήρων αποτελούν ένα είδος χρονοσειράς, αν και πρέπει να διαχωρίσουμε τις διαφορετικούς εξόδους των αισθητήρων
σε διαφορετικές απεικονίσεις των χρονοσειρών. Όλοι οι αισθητήρες με το ίδιο όνομα βρίσκονται στο ίδιο dashboard για μια συγκεκριμένη μετρική.
Έχουμε χωρίσει τις συγκεκριμένες εξώδεις που φαίνονται στο \ref{fig:grafana},αλλά μπορεί να προστεθεί οποιαδήποτε από τις εξόδους που έχουμε από τον κάθε αισθητήρα.

Το βασικό κομμάτι της Grafana που θα μας απασχολήσει είναι οι ψηφιακές πίνακες ελέγχου (Dashboard).
Ένα dashboard στο Grafana αποτελείται από πίνακες (panels) που απεικονίζουν δεδομένα μέσω γραφημάτων, διαγραμμάτων και άλλων μορφών οπτικοποίησης. Οι πίνακες αυτοί δημιουργούνται με τη χρήση επιμέρους συνιστωσών, οι οποίες μετασχηματίζουν τα ακατέργαστα δεδομένα που προέρχονται από μία πηγή δεδομένων σε οπτικές αναπαραστάσεις. 

Η διαδικασία αυτή περιλαμβάνει τη διέλευση των δεδομένων από τέσσερα διακριτά στάδια: ένα πρόσθετο λογισμικό (plugin) που λαμβάνει τα δεδομένα, ένα query που μας δίνει το κομμάτι των δεδομένων που μας ενδιαφέρει, έναν προαιρετικό μετασχηματισμό των δεδομένων(transformation) και τέλος την οπτικοποιήση σε ένα πίνακα (panel). 

Ένα plugin αποτελεί λογισμικό το οποίο επεκτείνει τις λειτουργικές δυνατότητες της Grafana. 
Τα πρόσθετα λογισμικού διακρίνονται σε διάφορες κατηγορίες, αλλά εμείς θα ασχοληθούμε για plugin πηγής δεδομένων (Data sources plugins).
 Ο ρόλος ενός Data sources plugins είναι να λαμβάνει τα δεδομένα που έλαβε έως απάντηση σε ένα querry που κάναμε,
 να ανακτά τα αντίστοιχα δεδομένα από την πηγή δεδομένων και να γεφυρώνει τις διαφορές μεταξύ του μοντέλου δεδομένων της πηγής δεδομένου και του μοντέλου δεδομένων των dashboards.

Η διαδικασία αυτή υλοποιείται μέσω μιας ενοποιημένης δομής δεδομένων, η οποία ονομάζεται πλαίσιο δεδομένων (data frame).
Ένα dataframe αποτελεί μία δομή δεδομένων που οργανώνει τα δεδομένα σε δύο διαστάσεις, σε γραμμές και στήλες, όπου οι γραμμές αποτελούν μία εγγραφή δεδομένων και οι στήλες αποτελούνται από τα πεδία των δεδομένων

Τα δεδομένα που εισέρχονται στο πρόσθετο από την πηγή δεδομένων ενδέχεται να βρίσκονται σε ποικίλες μορφές ,όπως JSON, μορφή γραμμών και στηλών ή αρχεία CSV.
Ωστόσο, κατά την έξοδό τους από το πρόσθετο και τη διέλευσή τους από τα επόμενα στάδια της διαδικασίας, με σκοπό την οπτικοποίησή τους, τα δεδομένα αναπαρίστανται πάντοτε υπό τη μορφή πλαισίων δεδομένων.

Τα queries επιτρέπουν τη μείωση του συνόλου των διαθέσιμων δεδομένων σε ένα συγκεκριμένο υποσύνολο δεδομένων, καθιστώντας την οπτικοποίησή τους περισσότερο διαχειρίσιμη. 
Σε περιπτώσεις όπου η μορφή των δεδομένων σε μια οπτικοποίηση δεν πληρεί τις απαιτήσεις, μπορεί να εφαρμοστεί transformation για την επεξεργασία των δεδομένων που παρέχονται από το queries.

Αφού τα δεδομένα εξαχθούν από την πηγή, περάσουν από το ανάλογο querry και του μετέπειτα μετασχηματισμού, περνούν στον πίνακα, στο τελικό. 
Ο πίνακας λειτουργεί ως δοχείο ή κοντέινερ (container  ) που εμφανίζει την οπτικοποίηση και παρέχει διάφορες ρυθμίσεις για την αλληλεπίδραση με αυτήν. 
Στη διαμόρφωση του πίνακα καθορίζεται ο τρόπος παρουσίασης των δεδομένων.

Στο δικό μας dashboard, χρησιμοποιήσαμε το infinity  data source plugin 
\url{https://grafana.com/docs/plugins/yesoreyeram-infinity-datasource/latest/},οπού τραβάει δεδομένα από κάποιο server που λειτουργεί με REST API 
και διαχειρίζεται διάφορες μορφές δεδομένων όπως JSON, CSV, GraphQL, XML, and HTML endpoints.
Αποτελεί μια καθολικά καλή εναλλακτική αν δεν υπάρχει κάποια εξειδικευμένο plugin για την πηγή δεδομένων που το θέλουμε. Υπάρχει plugin για την mongodb αλλά δεν είναι ελεύθερη προς χρήση.

Σαν querry, καταρχάς χρησιμοποιούμε σε όλες τις οπτικοποιήσεις την παρακάτω:

\begin{center}
\fbox{\detokenize{http://personal-laptop.local:8080/timeSeriesEndpoints/getTimeSeriesData}}\linebreak 
\fbox{\detokenize{?start=${__from}&end=${__to}&Id=N}}
Με N να είναι ο αριθμός id της συσκευής που θέλουμε τα δεδομένα
\end{center}

Οπού το \texttt{\detokenize{http://personal-laptop.local:8080/}} αποτελεί τον τρόπο 
να έχουμε πρόσβαση στον σέρβερ μας κα είναι σταθερό σε κάθε http αίτηση, 
το timeSeriesEndpoints/getTimeSeriesData αποτελεί το endpoints 
που στέλνει στον χρήστη τα δεδομένα από κάποια αρχική χρονική στιγμή έως κάποια τελική χρονική στιγμή.

Επιλεγούμε να πάρουμε τα δεδομένα από κάποιο συγκεκριμένο αισθητήρα ξεχωριστά .
Έπειτα, κρατάμε το timestamp (σημείωση χρόνου),που κρατάει τις χρονικές στιγμές των δεδομένων που επιλέχθηκαν,
όπως και τις τιμές του πεδίου που θέλουμε να απεικονίσουμε στον συγκεκριμένο πίνακα.
Δεν πραγματοποιούμε κάποιο μετασχηματισμό

Απεικονίζουμε σε κάθε διαφορετικά πίνακα ένα πεδίο μόνο, όπου για το συγκεκριμένο πεδίο έχουμε σαν διαφορετικές χρονοσειρές τις τιμές του κάθε αισθητήρα για την συγκεκριμένη πηγή.
Στα πειράματα μας, η συσκευή με \textbf{Id=0} είναι με μπλε χρώμα η χρονοσειρά,
η συσκευή  \textbf{Id=1} είναι με πράσινο και ο αισθητήρας \textbf{Id=0} είναι με κίτρινο.

Κατά την διάρκεια των πειραμάτων, εστιάσαμε κυρίως στους πίνακες με τα πεδία \textbf{Ακρίβεια} και \textbf{VOC από BME680},οπού αντιστοιχούν στις ψηφιακές εξόδους του BME680 \texttt{BME680:iaqAccuracy} 
και \texttt{BME680:breathVocEquivalent}.
